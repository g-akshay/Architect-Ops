# Architect-Ops: Context Builder Skill

## Purpose

When a new agent joins a project—in a fresh session, a new developer, or a new tool—it has no memory of how the system was designed. Context Builder creates institutional memory that any agent can read in seconds.

This is the skill that makes every future AI session smarter from the first message.

## When To Use

- When starting a project with Architect-Ops for the first time
- After a major sprint that added significant new business logic
- Before onboarding a new AI agent, tool, or developer
- After running `/debt-scanner` and `/migration-agent` (update the graph when the system changes)
- On a monthly cadence for large active codebases

## Activation Phrases

- "build context for this project"
- "update the knowledge graph"
- "map the system for me"
- "create KNOWLEDGE_GRAPH.md"
- "what does this codebase do?" (triggers context-builder if no graph exists)

## Output: KNOWLEDGE_GRAPH.md

The output is a markdown file at the project root. It is both machine-readable (for agents) and human-readable (for developers).

### KNOWLEDGE_GRAPH.md Structure

```markdown
# Knowledge Graph — [Project Name]
Last updated: [date]
Generated by: Architect-Ops /context-builder
─────────────────────────────────────────────

## Business Concepts

### [Concept Name]
- Owner module: [path/to/module]
- Dependents: [modules that use this concept]
- Data shape: [brief schema summary]
- Rules: [key business rules governing this concept]
- Notes: [tribal knowledge, edge cases, historical context]

## Module Map

| Module | Purpose | Owns | Depends On |
|--------|---------|------|------------|
| [path] | [what it does] | [concepts] | [other modules] |

## Data Flow

### [User Journey / Feature Name]
1. [Entry point] receives [input]
2. [Module A] [transforms/validates] it
3. [Module B] [stores/fetches] from [data source]
4. [Module C] [publishes/returns] [output]

## External Dependencies

| Package | Version | Used For | Owner Module | Notes |
|---------|---------|---------|-------------|-------|
| [name] | [ver] | [purpose] | [module] | [any restrictions] |

## Tribal Knowledge

> Tribal knowledge is anything that is true about the system but not obvious from reading the code alone.

- **[Topic]**: [explanation that would have saved a developer 2 hours]
- **[Topic]**: [known gotcha, edge case, or historical decision]

## Architectural Decisions Log

| Date | Decision | Reason | Alternatives Rejected |
|------|---------- |--------|----------------------|
| [date] | [what was decided] | [why] | [what was not chosen] |
```

## Execution Steps

### Step 1 — Scope Definition
Ask or infer:
- Root directory to analyze
- File types to include (default: all source files, exclude tests and generated files)
- Depth of analysis (module-level summary vs. function-level detail)

### Step 2 — Read Existing Context
If `KNOWLEDGE_GRAPH.md` exists, read it. This is an update run, not a rebuild.
Note what has changed since the last run (new files, deleted files, modified modules).

### Step 3 — Extract Business Concepts

For each module/feature directory:
- Identify the primary business concept it owns (User, Order, Payment, etc.)
- List the key operations (create, validate, transform, persist, publish)
- Note the data shapes flowing in and out
- Identify implicit contracts ("this module assumes X is always non-null")

### Step 4 — Build the Module Map

For each module:
- What does it do in one sentence?
- What business concepts does it own?
- What does it depend on from other modules?

### Step 5 — Trace Data Flows

For each major user-facing feature or system flow:
- Trace the path from entry point to final output
- Note where data is validated, transformed, persisted, and published
- Identify any async jumps (queue, event, background job)

### Step 6 — Extract Tribal Knowledge

Look for:
- Comments that explain *why*, not *what*
- Unusual logic with no associated comment (flag for the human to annotate)
- `// HACK`, `// FIXME`, `// NOTE` — these are tribal knowledge
- Conditionals guarding against edge cases that aren't obvious from the data model

### Step 7 — Write / Update KNOWLEDGE_GRAPH.md

- For a new graph: write the full document
- For an update: append new concepts, update changed modules, mark deprecated sections
- Always update the "Last updated" timestamp

## Important Rules for This Skill

1. Extract facts. Do not infer business rules that aren't explicit in the code.
2. If something is ambiguous, write it as a question in the Tribal Knowledge section.
3. Keep descriptions short. The goal is orientation, not documentation.
4. The graph should help a new agent bootstrap in under 5 minutes of reading.
5. Never overwrite the Architectural Decisions Log — only append to it.
